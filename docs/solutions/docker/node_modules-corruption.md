# Docker builds corrupting host node_modules

## Problem

After Docker builds, `yarn vitest run` (and `npm run test:unit`) fail with
`Error: Cannot find native binding` or module-not-found errors. Running
`find node_modules -user root` shows thousands of root-owned files.

Symptoms:
- `npm run test:unit` fails with rolldown/vitest binding errors
- `yarn vitest run` works when fresh-installed elsewhere (e.g. `/tmp` clone)
- The same project's tests pass in Docker (clean install inside container)

## Root Cause

The `.dockerignore` was missing standard Next.js entries. When Docker builds,
`COPY . .` copies the build context. If `node_modules/` isn't fully excluded
(or if past builds left root-owned files that got copied back through volume
mounts), the Docker daemon (running as root) creates root-owned files on the
host filesystem.

Yarn PnP is particularly sensitive because it uses hard links and symlinks
in `.yarn/cache/` and `.yarn/unplugged/` — root ownership breaks these.

## Solution

1. **Harden `.dockerignore`** with standard Next.js entries:
   ```
   node_modules/
   .yarn/cache/
   .yarn/unplugged/
   .yarn/install-state.gz
   .next/
   coverage/
   .env*
   ```

2. **Clean existing corruption** without sudo:
   ```bash
   docker run --rm -v "$(pwd)":/app -w /app alpine:latest \
     sh -c "chown -R $(id -u):$(id -g) node_modules"
   ```

3. **Reinstall if PnP state is stale:**
   ```bash
   rm -rf node_modules .yarn/cache .yarn/unplugged .yarn/install-state.gz
   yarn install
   ```

## Prevention

- `.dockerignore` must always exclude `node_modules/`, `.yarn/cache/`, and
  `.yarn/unplugged/`. These are recreated by `yarn install` inside the Docker
  build — they never need to be in the build context.
- After any `.dockerignore` change, run a test Docker build to verify the
  context size doesn't include these directories.
- The `ce-work` skill's Phase 0.5 health check catches this before any code
  is written.
